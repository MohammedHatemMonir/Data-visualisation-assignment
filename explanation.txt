🧠 Detailed Code Justification (Section by Section)
Step 1 — Data Loading & Inspection
The code loads the full 30,056-row CSV first, then prints shape, dtypes, statistics, and missing counts before making any preprocessing decision. This is the correct scientific order — you can't choose a strategy for missing values without knowing how many exist and why.
Key finding: device_name and device_location have exactly 9,041 missing values — the same count. This tells us it's systematic (not random), so these two columns go missing together, likely because some device packets skip metadata.

Step 2 — Preprocessing (4 sub-steps)
2a — Timestamp parsing + cyclical hour encoding:
Raw ISO timestamps (2026-02-09T07:16:19+00:00) can't be used by a model directly. We extract hour, month, and day_of_week. Crucially, hour is encoded as sin(2π·h/24) and cos(2π·h/24) — this is called cyclical encoding. Without it, a model would think hour 23 and hour 0 are far apart (difference of 23), when really they are adjacent. The sin/cos pair fixes this mathematically.
2b — Imputing missing device_name:
Rather than dropping 9,041 rows (~30% of data), we build a device_id → device_name lookup from non-null rows. Since every device_id appears in both null and non-null rows, we can recover the name 100%. Dropping rows would be wasteful and introduce bias if some devices are more likely to have missing metadata.
2c — Outlier capping (IQR method):
Temperature max was 54.33°C, which is physically impossible for outdoor sensors in Victoria, Australia (record is ~48°C). We cap at Q3 + 1.5×IQR = 32.56°C. We chose IQR over Z-score because the temperature distribution is right-skewed — Z-score assumes normality, and on skewed data it misidentifies too many normal values as outliers. IQR is a non-parametric, robust method.
2d — Feature engineering:
humidity_sq = humidity² captures the non-linear relationship between humidity and temperature. The scatter plot (EDA) shows humidity and temperature don't follow a straight line. Adding the square gives linear models a way to capture this curve. The interaction term temp × humidity was created for EDA but excluded from the final model to avoid multicollinearity.

Step 3 — Heterogeneous Data Integration
The assessment requires data from "at least one other relevant source." The code calls the Open-Meteo API (a free, open-access ERA5 reanalysis service) to fetch daily apparent temperature max/min, wind speed, and precipitation for the Victoria, AU location. This is merged on date via a left join — original rows are always kept even if a date has no API match. The code also has a graceful fallback if the API is unavailable (the sandbox blocked it, but on GitHub/your machine it will work perfectly).

Step 5 — Train/Test Split
80/20 split with random_state=42 for reproducibility. StandardScaler is applied to Linear and Ridge (required — these models are sensitive to feature scale; a humidity value of 76% and humidity_sq of 5,776 are on completely different scales without normalisation). Random Forest is scale-invariant so scaling isn't required, but the pipeline keeps it consistent.

Step 6 — Three Regression Models
Linear Regression: The mathematical baseline. Fits y = β₀ + β₁x₁ + ... + βₙxₙ by minimising sum of squared errors analytically. Completely interpretable. Assumes a linear relationship — we partially satisfied this by adding humidity_sq.
Ridge Regression (α=1.0): Adds L2 penalty λΣβᵢ² to the OLS loss function. This shrinks all coefficients slightly, which is important here because humidity_merged and humidity_sq have correlation r ≈ 0.94 (multicollinearity). High multicollinearity makes OLS coefficient estimates unstable/high-variance. Ridge fixes this at the cost of a tiny bias. In practice the results were nearly identical to OLS here, which tells us multicollinearity wasn't severe enough to matter much.
Random Forest: An ensemble of 100 decision trees, each trained on a bootstrapped sample with a random feature subset at each split. Captures non-linear interactions between humidity, hour, and temperature without needing to manually specify them. Its R² of 0.803 vs 0.687 for linear models shows there are real non-linear patterns in this data.

Step 8 — Paired t-tests
Each model was evaluated on the same 5 folds (KFold, n=5). A paired t-test compares fold-level RMSE scores between two models. It's "paired" because fold 1's data is the same for both models — removing data variation from the test. Results: Linear ≈ Ridge (p = 0.91, not significant), but Random Forest beats both (p ≈ 0.000, highly significant).
